# CLAUDE.md

給 Claude Code 的專案常駐說明。動工前請先讀本檔,以及 `docs/design/` 裡與當前任務相關的設計文件。

---

## 專案定位

一個**偽裝成復古匿名論壇(無名小站／PTT 那個年代)的解謎 ARG 網站**。玩家在論壇上跟著一個沉寂帳號發起的「懸賞尋寶」逐關解謎,最後拼出一樁被權勢者掩蓋的死亡真相,並在終局做出「揭發」或「收封口費」的選擇。

調性:**心理驚悚、真假難辨**。核心手法是每條線索同時支持「死者的鬼在發文」與「有人在做局」兩種解讀,遊戲**從不解答是哪一種**。

## 技術棧與硬限制

- **框架**:Vue 3(Composition API,`<script setup>`)+ Vite
- **樣式**:Tailwind CSS。復古 BBS 視覺(等寬字、深色終端感、極簡)集中在一套共用元件與 Tailwind 設定裡,不要每個頁面重寫樣式
- **路由**:Vue Router。隱藏頁走不公開的路由(見下方「解謎實作」)
- **部署**:Vercel(靜態 SPA)。**不要引入後端或資料庫**——這是純前端專案
- **狀態**:玩家進度用 Pinia 或簡單的 composable 管理即可,不要過度設計
- 不要為了幾個隱藏頁去搭 server;所有解謎邏輯都在前端完成

## 檔案結構

```
├─ CLAUDE.md              ← 本檔
├─ docs/
│  ├─ design/             ← 關卡設計與劇情(唯一內容來源,動工前先讀)
│  └─ 數字密碼總表.md      ← 所有日期/座標/密碼答案的單一事實來源
├─ src/
│  ├─ components/         ← 共用 BBS 元件(文章串、推文、個資卡、站內信…)
│  ├─ views/             ← 各頁型與各關卡頁面
│  ├─ router/            ← 含隱藏路由
│  ├─ stores/            ← 玩家進度
│  ├─ data/              ← 貼文/推文/報告等內容資料(與畫面分離)
│  └─ assets/           ← 圖片線索(含調亮版)
└─ ...
```

## 設計文件在哪、怎麼用

**完整的關卡設計、劇情、謎題解答都在 `docs/design/`。** 動工任何一關前,先讀該關對應的那份 `.md`,不要憑印象實作。

各關數字(密碼、路由編號、日期、座標)**一律以 `docs/design/數字密碼總表.md` 為準**。若某份設計文件的數字與總表不一致,以總表為準,並提醒我更新。

## 這個專案的鐵則

1. **數字單一來源**:所有密碼/路由/日期/座標只在 `數字密碼總表.md` 定義一次,程式碼引用它,不要在各關硬寫、各自編號,以免兜不攏。
2. **敏感題材維持留白**:本作觸及未成年受害、家庭創傷、物質濫用。設計文件已刻意用暗示與留白處理。實作時**照原文,不要補寫任何細節**(不寫毒品種類/劑量/用法、不寫犯罪或死亡過程)。若某段內容需要擴寫,先問我。
3. **內容與畫面分離**:貼文、推文、報告等文字放 `src/data/`,元件負責渲染。方便校對與埋線索。
4. **真假難辨不破梗**:程式碼、註解、變數命名都不要在前端明示「他是鬼」或「這是某人做的局」。玩家看 source 是玩法一部分,但不該直接撿到答案。
5. **虛構聲明**:頁尾或 about 頁固定放一行「本作為虛構解謎創作」,避免仿真內容被誤當真。人物、政黨、案件全架空,不影射真實對象。

## 解謎實作備忘

- 隱藏頁:用不公開的路由(例:`/thread/178`),不掛在任何導覽列。前端驗證即可。
- 密碼關:錯誤時回一句 in-world 的話(見各關設計),別用生硬的「密碼錯誤」。
- 「深夜才出現」的內容:JS 讀使用者本地時間判斷。
- 圖片調亮線索:CSS `filter: brightness()` 或另存一張亮版放 `assets/`。
- EXIF 線索:可做成頁面上模擬的「相片資訊」面板(照顧一般玩家),進階版再把資料寫進真圖檔。

## 開發流程偏好

- **一次做一關/一個聚焦任務**,不要一次把整站做完。用設計文件當規格逐關餵。
- 先做共用 BBS 外殼與版面模板,之後每關套用。
- 每完成一個可運作的段落就 commit,commit 訊息講清楚做了什麼。
- 改動前若牽涉多個檔案或既有結構,先說明計畫再動手。
